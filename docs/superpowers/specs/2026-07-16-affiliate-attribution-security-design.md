# 分销归因安全优化设计

## 目标

在保持现有 `/r/:affCode`、邮箱注册和 OAuth 注册兼容性的前提下，提高推广入口抗刷能力、真实 IP 可信度和点击到注册的归因准确性。

## 设计

1. 推广入口使用现有 Redis 原子限流器，增加可配置键函数，并依次限制可信客户端 IP、IP+User-Agent 指纹和邀请码。Redis 故障时对推广入口采用 fail-close，避免故障期间被无限写入。
2. 只通过 Gin `SetTrustedProxies` 解析转发地址。未配置可信代理时忽略 `X-Forwarded-For` 和 `CF-Connecting-IP`；Cloudflare/Nginx 的 CIDR/IP 由部署配置显式提供。
3. 成功访问推广入口后签发 HttpOnly、SameSite=Lax、短期有效的 HMAC 签名 Cookie。载荷包含访问事件 ID、邀请码和过期时间，不保存原始 IP 或 User-Agent。
4. 全局归因中间件验证 Cookie，并将可信访问事件写入请求 context。邮箱及 OAuth 注册沿用现有 context；邀请绑定事务仅在事件邀请码与邀请人一致时把访问事件关联到新用户。
5. 新增注册归因字段与索引，不允许同一用户重复归因。无 Cookie、失效 Cookie 或篡改 Cookie 不阻断注册，仍兼容直接填写邀请码，但不会被计入“已验证点击注册”指标。

## 错误与安全边界

- 无效/未授权邀请码返回 404，避免枚举。
- 超过限流返回 429。
- Cookie 验证失败静默忽略，不泄露验证细节。
- 数据库只保存 HMAC 访客摘要、签名事件 ID和聚合维度，不保存原始 IP、完整 User-Agent 或 Cookie。
- 可信代理列表为空是安全默认值；生产部署必须按实际反向代理配置精确 CIDR/IP。

## 验收标准

- 伪造转发头在无可信代理配置时不能改变限流/访客 IP。
- IP、指纹、邀请码任一维度超限均返回 429。
- Cookie 篡改或过期不能产生已验证归因。
- 有效 Cookie 注册后，访问事件在绑定事务内关联注册用户。
- 现有注册、OAuth、返利、冻结和站内余额结算行为保持兼容。
- Go 全量测试、嵌入式生产测试、前端构建、Docker 健康检查和运行态回归通过。
